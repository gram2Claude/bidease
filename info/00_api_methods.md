# Сводка методов API — Bidease Reporting

Источник правды: официальная документация
https://support.bidease.com/support/solutions/articles/72000586529-bidease-reporting-api-documentation
(изучена 2026-07-21). Поля и типы — оригинальные, как в доках, без перевода.

Bidease — mobile DSP. Reporting API — **один синхронный GET-эндпоинт**, отдающий **CSV**.
Справочников-эндпоинтов (кампании/креативы/продукты) нет — атрибуты сущностей доступны
только как группировки отчёта (`CampaignID`, `CampaignName`, `CreativeID`, …).

---

## 1. Аутентификация — выпуск и обновление токенов

**Эндпоинтов выпуска/обновления токена НЕТ.** Используется **статический API-токен**,
который выдаёт техподдержка или CSM Bidease. TTL не документирован (токен долгоживущий,
обновление — только ручной перевыпуск через поддержку).

- **Способ передачи:** query-параметр `api-token` в каждом запросе (не заголовок).
  ⚠️ Написание канонично **через дефис** — `api-token` (так в таблице параметров и в
  verbatim-примере доков; в одном из примеров статьи встречается `api_token` — при
  smoke-тесте проверить, принимается ли только дефисное или оба написания).
- **Привязка токена:** к токену привязана **таймзона** (`API_TOKEN timezone`, по умолчанию
  UTC+0) — все даты/часы запроса и ответа интерпретируются в ней. Область видимости
  (какие рекламодатели/кампании доступны токену) в доках не описана — установить фактом
  при smoke-тесте.

## 2. Методы получения статистики

### Метод: `GET https://ui-api.bidease.com/api/reporting/v1/stats`

- **Название в доках:** Bidease Reporting API
- **Назначение:** статистика (расход, показы, клики, установки, доход, IAP, цели)
  с произвольной группировкой до 7 разрезов
- **Тип:** синхронный (ответ сразу в теле, без submit/poll/download)
- **Группировка / разрезы:** параметр `group` (повторяемый, **максимум 7**); без `group` —
  один суммарный ряд за период

- **Параметры запроса (оригинальные имена; все — query string):**

  | Поле | Тип | Обязательное | Описание / допустимые значения |
  |------|-----|-------------|-------------------------------|
  | `api-token` | string | да | API-токен (выдаёт поддержка/CSM) |
  | `fromdate` | string | да | Начало периода, **включительно**. `YYYY-MM-DD` или относительная дата |
  | `todate` | string | да | Конец периода, **ЭКСКЛЮЗИВНО** (день `todate` в отчёт НЕ входит). `TO_DATE` должен быть больше `FROM_DATE` |
  | `group` | string | нет | Разрез отчёта; повторяемый, ≤ 7 значений; список ниже |
  | `group-by-param` | string | нет | Кастомные биндинги (напр. `CustomCampaign`) |
  | `attr-type` | string | нет | Атрибуция времени: `ByEvent` (дефолт) / `ByImpression` / `ByConv` |
  | `rekls` | integer | нет | Фильтр по ID рекламодателя; повторяемый |
  | `products` | integer | нет | Фильтр по ID продукта; повторяемый |
  | `campaigns` | integer | нет | Фильтр по ID кампании; повторяемый |
  | `bundles` | string | нет | Фильтр по бандлу приложения; повторяемый |
  | `extra-column` | string | нет | Доп. вычисляемые колонки (значения — из Bidease UI: `UniqClicks`, `UniqImpressions`, `Ecpa4`, …); повторяемый |
  | `noheader` | boolean | нет | `true` — не выводить строку заголовка CSV (дефолт `false`) |

- **Относительные даты:** формат `^(today)(-\d+d)?$` — `today`, `today-20d`.
  `today` = начало текущего дня **в таймзоне токена** на момент вызова.
  Пример «за вчера»: `&fromdate=today-1d&todate=today`.

- **Значения `group` (все 26, verbatim из доков):**

  | Значение | Описание |
  |----------|----------|
  | `Day` | день события |
  | `Hour` | час события (в таймзоне токена) |
  | `ImpressionDay` | день показа |
  | `ImpressionHour` | час показа |
  | `ConvDay` | день конверсии (инсталла) |
  | `ConvHour` | час конверсии (инсталла) |
  | `Country` | страна (ISO-3) |
  | `Region` | регион |
  | `City` | город |
  | `Make` | производитель устройства |
  | `Model` | модель устройства |
  | `DeviceType` | тип устройства (Mobile, Tablet) |
  | `OperationSystem` | операционная система |
  | `OperationSystemVersion` | версия ОС |
  | `HasIFA` | передан ли рекламный идентификатор устройства |
  | `CampaignID` | ID кампании |
  | `CampaignName` | название кампании |
  | `CreativeID` | ID креатива |
  | `CreativeHash` | уникальный хеш ресурсов креатива |
  | `PlacementID` | ID плейсмента в приложении |
  | `Application` | бандл приложения, где был показ |
  | `Carrier` | оператор связи |
  | `Exchange` | название биржи (exchange) |
  | `AdvertiserID` | ID рекламодателя на ui.bidease.com |
  | `ProductID` | ID продукта на ui.bidease.com |
  | `Currency` | валюта платежа (для payment-событий) |

- **Формат ответа:** `text/csv` (единственный формат; JSON нет)
- **Кодировка:** не документирована (ожидается UTF-8 — зафиксировать фактом при smoke-тесте)
- **Разделитель:** не документирован (ожидается `,` — парсится стандартным CSV-парсером,
  в примере интеграции доков используется `Utilities.parseCsv()`); зафиксировать фактом
- **Структура ответа — колонки CSV (метрики, в исходном порядке):**

  | № | Колонка (оригинал) | Тип | Описание |
  |---|-------------------|-----|----------|
  | 1 | `spend` | number | медийный расход **в долларах** |
  | 2 | `impressions` | integer | показы |
  | 3 | `clicks` | integer | клики |
  | 4 | `conversions` | integer | конверсии (**инсталлы**) |
  | 5 | `revenue` | number | доход, переданный из MMP |
  | 6 | `iap_sum` | number | сумма внутриигровых покупок |
  | 7 | `iap` | integer | количество внутриигровых покупок |
  | 8–13 | `goal1` … `goal6` | integer | количество in-app-событий (цели из Bidease UI) |
  | 14 | `ctr` | number | click-through rate (расчётная) |
  | 15 | `ecpm` | number | эффективная цена за 1000 показов (расчётная) |
  | 16 | `ecpc` | number | эффективная цена клика (расчётная) |
  | 17 | `i2c` | number | отношение показов к конверсиям (расчётная) |
  | 18 | `cpi` | number | цена за инсталл (расчётная) |

  **После метрик** — колонки группировок **в порядке их указания в запросе**.
  Первая строка — заголовок (если не передан `noheader=true`).

- **Особенности / ограничения:**
  - Максимальный диапазон дат: **в пределах 1 года от текущей даты**
  - `todate` — **эксклюзивная** граница (⚠️ главная ловушка API: «по 20-е включительно» = `todate=21-е`)
  - Все даты/часы — **в таймзоне токена** (по умолчанию UTC+0)
  - Rate limits: **не документированы**
  - Пагинации нет — весь результат одним CSV-телом
  - Ошибки/HTTP-коды: **не документированы** — поведение (невалидный токен, кривой период)
    установить фактом при smoke-тестах и зафиксировать здесь
  - Поведение при пустом результате: не документировано (ожидается пустой CSV/только заголовок) — проверить фактом

- **Пример запроса (verbatim из доков):**

  ```
  https://ui-api.bidease.com/api/reporting/v1/stats?api-token={API_TOKEN}&fromdate={FROM_DATE}&todate={TO_DATE}&rekls={REKL}&products={PRODUCT}&campaigns={CAMPAIGN}&group={GROUP1}&group={GROUP2}
  ```

  Пример с группировками (даты фиксированные):

  ```
  https://ui-api.bidease.com/api/reporting/v1/stats?api-token=XXX&fromdate=2019-12-01&todate=2020-01-01&rekls=300&rekls=310&group=Application&group=Country&group=CreativeID&group=CampaignID
  ```

## 3. Сводная таблица методов статистики

| # | Endpoint | Назначение | Формат ответа | Sync/Async | Ключевые поля ответа |
|---|----------|------------|---------------|------------|---------------------|
| 1 | `GET /api/reporting/v1/stats` | Универсальный отчёт с группировками (≤7) | `text/csv` | sync | `spend`, `impressions`, `clicks`, `conversions`, `revenue`, `iap_sum`, `iap`, `goal1–6`, `ctr`, `ecpm`, `ecpc`, `i2c`, `cpi` + колонки группировок |

Это **единственный** метод Reporting API — все проектные функции (справочник кампаний,
дневная статистика по кампаниям/креативам, admin_audit) строятся на нём разными
комбинациями `group`/фильтров/периодов.

- Bidease — Singular Integration
